# SVETOZAR SCHOOL — Архитектура и устойчивость к изменениям

**Версия:** 1.0 · **Цель документа:** так организовать структуру и код, чтобы правки и доработки **почти не могли что-то сломать**, а если что-то сломано — это ловится автоматически до попадания в проект.

---

## 0. Главная идея

Сломать код «случайно» можно по трём причинам: (1) правка в одном месте незаметно ломает другое, (2) приходят неожиданные данные, (3) ошибку никто не замечает до продакшена. Поэтому защита строится на трёх уровнях:

1. **Изоляция** — код разбит на независимые модули-фичи. Правка в «уроках» не может задеть «оплату доступов» или «рейтинг», потому что они не знают друг о друге напрямую.
2. **Контракты и типы** — TypeScript + Zod + Prisma. Несовпадение типов = ошибка **до запуска**, а не баг в проде.
3. **Автоматические ворота (guardrails)** — линтер, типы, тесты и CI блокируют любой коммит/мерж, который что-то ломает. Это и есть «невозможно сломать»: сломанный код просто не проходит дальше.

Принцип: **легко добавить новое, трудно сломать старое.**

### Анатомия одной фичи (на примере `features/lessons/`)

```
features/lessons/
├─ schema.ts        # Zod-схемы входов/выходов (контракт данных)
├─ service.ts       # бизнес-логика (только тут «как работает»)
├─ queries.ts       # доступ к БД через Prisma (только тут SQL/ORM)
├─ actions.ts       # server actions — тонкая обёртка: валидация → service
├─ components/      # UI этой фичи (LessonPlayer, LessonList…)
├─ hooks.ts         # клиентские хуки этой фичи
└─ index.ts         # ★ публичный API модуля — наружу торчит ТОЛЬКО он
```

**Правило `index.ts`:** другие части проекта импортируют фичу только через её `index.ts`. Внутренности (`queries`, `service`) — приватные. Поэтому можно как угодно менять внутренности урока, не боясь задеть остальной код — снаружи виден только стабильный «фасад».

## 2. Правила зависимостей (направление импортов)

Зависимости текут строго в одну сторону — это исключает запутанные циклы и «эффект домино»:

```
app/ (роуты)  →  features/ (логика)  →  lib/providers, lib/db, config/
components/ui ←  переиспользуются всеми, но сами ни от кого не зависят
```

- Роут (`app/`) **ничего не вычисляет** — только вызывает action из фичи и рендерит.
- Фича **не лезет** в другую фичу напрямую — только через её `index.ts` (а лучше через события/общие сервисы).
- UI-компоненты в `components/ui/` **не содержат бизнес-логики** — их можно менять без риска для логики.

Соблюдение направления проверяется автоматически (ESLint-правило `import/no-restricted-paths`), так что нарушить случайно не выйдет.

## 3. Контракты и типы — ошибка ловится до запуска

- **TypeScript `strict: true`** — несовпадение типов не компилируется. Переименовал поле в одном месте — компилятор покажет все места, где надо поправить.
- **Prisma как единый источник правды о данных:** схема БД → автоматически генерируются типы. Нельзя обратиться к несуществующему полю.
- **Zod на всех границах:** любые данные извне (формы, API, вебхуки Telegram, ответ ИИ) проходят через Zod-схему. Кривые данные отсекаются на входе, а не ломают логику глубже.
- **Один тип — одно определение.** Типы выводятся из Zod/Prisma, не дублируются руками.

Эффект: 80% «поломок» в JS-проектах — это `undefined is not a function` из-за неожиданных данных. Тут таких классов ошибок просто нет.

## 4. Централизация (меняем в одном месте)

Всё, что может меняться, живёт в одном месте, а не разбросано по коду:

- `config/constants.ts` — все числа геймификации (XP за урок/модуль/ДЗ, проходные баллы, лимиты попыток, rate-limits). Поменять баланс = поправить одну строку.
- `config/env.ts` — список и валидация переменных окружения.
- `styles/tokens` — цвета, шрифты, радиусы (золото/чёрный). Сменить оттенок золота = одно значение.
- `prisma/schema.prisma` — структура данных.

Магических чисел и строк, «вшитых» по коду, — нет.

---

## 5. Абстракции внешних сервисов (замена без боли)

Внешние сервисы спрятаны за интерфейсами в `lib/providers/`. Остальной код знает только интерфейс, не конкретный сервис:

```ts
// lib/providers/video.ts
export interface VideoProvider {
  getSignedUrl(lessonId: string, userId: string): Promise<string>;
}
// сегодня внутри — Bunny; завтра можно заменить, не трогая ни одного компонента
```

Так же `AiProvider` (Claude) и `Notifier` (Telegram/email). Если завтра меняется видеохостинг или модель ИИ — правка в одном файле-адаптере, весь остальной код не замечает.

## 6. Контейнеризация ошибок (одна поломка ≠ падение всего)

- **Server actions возвращают типизированный результат** `{ ok: true, data } | { ok: false, error }`, а не кидают исключения наружу. Вызывающий код всегда знает оба исхода.
- **React Error Boundaries** вокруг крупных блоков (плеер, чат наставника, рейтинг). Если упал один виджет — остальная страница работает.
- **Деградация вместо краха:** проверка ДЗ недоступна → ответ сохраняется со статусом «на проверке» (уже в ТЗ). Видео не грузится → показываем сообщение, не белый экран.
- Все внешние вызовы (ИИ, Bunny, Telegram) — в `try/catch` с понятным фолбэком.

---

## 7. Валидация окружения при старте

`config/env.ts` проверяет все переменные Zod-схемой при запуске. Если ключа нет или он кривой — приложение **падает сразу с понятной ошибкой**, а не «работает странно» и роняет фичу в проде. Это превращает класс «тихих» конфиг-багов в громкие и мгновенные.

---

## 8. Автоматические ворота — то, что реально «не даёт сломать»

Это ключевой механизм. Проверки запускаются **на каждый коммит и перед мержем**; не прошёл — код не попадает в проект.

| Проверка | Что ловит | Когда |
|---|---|---|
| `tsc --noEmit` (типы) | несовпадение типов, опечатки в полях | pre-commit + CI |
| ESLint | плохие паттерны, нарушение границ импортов | pre-commit + CI |
| Prettier | форматирование (чтобы диффы были чистыми) | pre-commit |
| Unit-тесты | сломанная бизнес-логика (XP, проверка ДЗ, прогресс) | pre-commit + CI |
| Integration/E2E | сломанные сценарии (вход, прохождение урока, сдача ДЗ) | CI |
| gitleaks | случайно закоммиченный секрет | pre-commit + CI |
| Prisma migrate check | рассинхрон схемы и БД | CI |

Инструменты: **Husky + lint-staged** (локальные хуки) и **GitHub Actions** (CI на сервере). Ветка `main` защищена: мерж только при зелёном CI.

**Тесты в первую очередь покрывают критичный путь:** начисление XP/стриков, логику проверки ДЗ и фолбэк, контроль доступа (ownership — нельзя открыть чужой урок), разблокировку следующего урока. Эти места ломать дороже всего — поэтому они под тестами.

## 9. Как вести разработку с Claude Code, чтобы не ломать

- **`CLAUDE.md` в корне** описывает правила: структура фич, направление импортов, «логика — в service, не в роутах», «секреты — только server-side», «новые числа — в constants». Claude Code держит это в контексте и пишет код в едином стиле.
- **Маленькие изменения.** Одна фича/правка за раз, отдельная ветка → прогон проверок → мерж. Маленький дифф = маленький риск и понятный откат.
- **Перед каждым мержем — `npm run check`** (типы + линт + тесты одной командой). Зелёно — мержим, красно — чиним.
- **Аддитивные миграции БД.** Новые поля/таблицы добавляем, старое не ломаем; удаление — отдельным осознанным шагом. Так данные и старый код не падают.
- **Фичефлаги** для крупного нового функционала: выкатываем выключенным, включаем, когда готово, мгновенно выключаем при проблеме — без отката кода.
- **Git = страховка.** Любая правка обратима одним откатом коммита.

## 10. Чеклист «правка не сломает проект»

- [ ] Изменение внутри одной фичи и не торчит за её `index.ts`?
- [ ] Типы проходят (`tsc`)? Линтер чист?
- [ ] Новые данные извне валидируются Zod?
- [ ] Числа/строки добавлены в `constants`/`config`, а не вшиты?
- [ ] Внешний вызов обёрнут в try/catch с фолбэком?
- [ ] Миграция аддитивная (ничего не удаляет молча)?
- [ ] Критичная логика покрыта тестом?
- [ ] Секреты не попали в клиент и в git?
- [ ] CI зелёный перед мержем?

Если все галочки стоят — сломать что-то крайне сложно. Если хотя бы одна не стоит — ворота из §8 не пропустят.

---

### Итог
«Невозможность сломать» — это не аккуратность руками, а **система**: изолированные фичи (нельзя задеть лишнее) + строгие типы и валидация (нельзя передать кривое) + автоматические проверки (нельзя смержить сломанное). Добавлять новое в такой структуре легко и безопасно; ломать старое — система не даёт.

# CLAUDE.md — правила работы над проектом SVETOZAR SCHOOL

> Этот файл читает Claude Code перед каждой задачей. Он держит код в единой устойчивой структуре, описанной в `ARCHITECTURE.md`. Следуй ему всегда.

## Что это за проект
SVETOZAR SCHOOL — обучающая платформа (LMS): курсы → модули → уроки, видео через собственный плеер, домашние задания с проверкой ИИ-наставником, геймификация (XP, стрики, достижения, рейтинг), профиль, центр знаний, интеграция с Telegram и админ-панель. Аудитория — до ~150 учеников по всему миру. Подробности — в `TZ_LMS_Platform.md`.

## Стек
- **Next.js (App Router) + TypeScript** (`strict: true`)
- **PostgreSQL + Prisma ORM**
- **Auth.js (NextAuth)** — вход по email+пароль, роль по умолчанию `student`
- **Tailwind CSS + shadcn/ui**, дизайн-токены (чёрный + золото) в `src/styles/tokens`
- **Anthropic SDK** — проверка ДЗ (только server-side)
- **Bunny Stream** — видео и файлы (подписанные ссылки)
- **Telegram Bot API (grammY)** — оповещения и результаты ДЗ
- **Zod** — валидация на всех границах

## Железные правила структуры (не нарушать)
1. **Логика живёт в `features/<имя>/service.ts`.** Роуты в `app/` — тонкие: вызвать action и отрендерить. Никакой бизнес-логики в роутах и компонентах.
2. **Доступ к БД только в `features/<имя>/queries.ts`** через Prisma. Нигде больше Prisma не импортируется напрямую.
3. **Наружу фича отдаёт только `features/<имя>/index.ts`.** Внутренности (`service`, `queries`) приватны. Импортировать чужую фичу — только через её `index.ts`.
4. **Направление зависимостей:** `app/ → features/ → lib/providers, lib/db, config/`. Фича не лезет в другую фичу напрямую. `components/ui/` ни от кого не зависит.
5. **Внешние сервисы — только через абстракции** в `lib/providers/` (`video.ts`, `ai.ts`, `notify.ts`). Не вызывай Bunny/Claude/Telegram напрямую из фич.

## Правила, защищающие от поломок
- **Валидируй любые входные данные Zod-схемой** (формы, API, вебхуки, ответ ИИ) в `features/<имя>/schema.ts`. Никаких «сырых» данных в логике.
- **Server actions возвращают** `{ ok: true, data } | { ok: false, error }`, не кидают исключения наружу.
- **Внешние вызовы** (ИИ, Bunny, Telegram) — в `try/catch` с фолбэком (например, ДЗ при недоступном ИИ сохраняется со статусом «на проверке»).
- **Все числа и лимиты** (XP, проходные баллы, попытки, rate-limits) — только в `config/constants.ts`. Не вшивай магические числа в код.
- **Миграции БД только аддитивные.** Не удаляй/не переименовывай поля без отдельного осознанного шага.
- **Типы выводи** из Prisma/Zod, не дублируй руками.

## Безопасность (критично)
- **Секреты — только server-side.** Никаких ключей в клиентских компонентах и без префикса `NEXT_PUBLIC_`. Вызовы Claude/Bunny/Telegram — из серверного кода.
- **Контроль доступа на каждом защищённом действии:** проверяй роль и владение ресурсом (ownership) — ученик не должен открыть чужой урок/ДЗ или урок без `enrollment`. Проверка — на сервере, не только в UI.
- **Роль `admin`/`curator` нельзя получить через регистрацию** — только вручную. Новые аккаунты = `student`.
- **Видео и файлы** отдаются по подписанной ссылке с коротким TTL и проверкой доступа, не прямым URL.
- **В Claude API не передавай** email/имя ученика — только текст ДЗ + база знаний. Защита от prompt injection: пользовательский текст — как данные, не как инструкции; результат строго парсится как JSON.
- **Не логируй** пароли, токены, полные email, тексты ДЗ.
- Пароли — argon2id. Сессии — HttpOnly+Secure cookie.

## Рабочий процесс
- **Одна задача — одна ветка — маленький дифф.** Не делай несвязанные правки в одном изменении.
- **Перед завершением задачи запусти `npm run check`** (типы + линт + тесты). Не оставляй красное.
- **Критичную логику покрывай тестом** (XP/стрики, проверка ДЗ и фолбэк, ownership/доступ, разблокировка уроков).
- **Крупный новый функционал — за фичефлагом**, выкатывай выключенным.

```
npm run check        # tsc + eslint + tests (запускать перед мержем)
npm run lint         # только линтер
npm run typecheck    # только типы
npm run test         # тесты
npx prisma migrate dev --name <name>   # новая аддитивная миграция
npx prisma studio    # просмотр БД
```

## Definition of Done (задача готова, если…)
- [ ] изменение внутри одной фичи, наружу — через `index.ts`;
- [ ] входные данные валидируются Zod;
- [ ] числа/строки в `config`, не вшиты;
- [ ] внешние вызовы обёрнуты в try/catch с фолбэком;
- [ ] проверены роль и ownership на сервере;
- [ ] секреты не попали в клиент/логи/git;
- [ ] миграция аддитивная;
- [ ] критичная логика под тестом;
- [ ] `npm run check` зелёный.

## Чего НЕ делать
- Не писать бизнес-логику в роутах/компонентах.
- Не импортировать Prisma вне `queries.ts`.
- Не обращаться к внешним сервисам мимо `lib/providers/`.
- Не вшивать секреты и магические числа.
- Не делать разрушительные миграции «по-тихому».
- Не лезть в чужую фичу мимо её `index.ts`.

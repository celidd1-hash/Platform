# Техническое задание: образовательная платформа (LMS)

**Версия:** 1.7 (MVP) · **Дата:** 12.06.2026
**Рабочее название:** SVETOZAR SCHOOL
**Разработка:** VS Code + Claude Code

---

## 1. Цель проекта

Веб-платформа для размещения авторских курсов и обучения учеников: просмотр видеоуроков, написание конспектов по усвоенному материалу с ИИ-проверкой, система прогресса и рейтинг учеников.

**Масштаб MVP:** 100–150 учеников **по всему миру** (важно: CDN-доставка видео и низкая задержка в разных регионах), 1–5 курсов. Монетизация на платформе отсутствует — доступы выдаёт админ. Архитектура должна позволять рост без переписывания.

---

## 2. Роли пользователей

| Роль | Возможности |
|---|---|
| **Ученик** | Регистрация/вход по email, просмотр доступных курсов, прохождение уроков, написание конспектов, просмотр своего прогресса, рейтинга и достижений |
| **Админ (владелец)** | Всё, что ученик + создание/редактирование курсов, модулей, уроков; управление учениками; выдача доступа к курсам; просмотр конспектов и статистики |
| **Куратор** *(фаза 2)* | Просмотр прогресса и конспектов учеников своей группы, без права редактировать курсы |

## 3. Функциональные требования

### 3.1 Аутентификация
- Регистрация и вход по **email + пароль**. Email = логин (уникальный идентификатор ученика).
- Подтверждение email при регистрации (ссылка на почту).
- Восстановление пароля через email.
- Сессии через JWT/cookie, защищённые маршруты по ролям.
- *(Опционально, фаза 2)* вход через Google OAuth.

### 3.2 Каталог курсов
- Главная страница: сетка карточек курсов (референс — onai.academy/courses).
- Карточка курса: обложка/иконка, название, краткое описание, длительность (недели), количество модулей и уроков, рейтинг курса, список ключевых тем (4–5 буллетов), статус для текущего ученика:
  - **«Начать обучение»** — доступ открыт;
  - **«Продолжить»** — курс начат (показывать % прогресса);
  - **«Недоступен» / «Приобрести»** — доступа нет (бейдж с замком).
- Доступ к курсу выдаёт админ вручную (enrollment). Оплата происходит вне платформы; модель данных закладываем так, чтобы в фазе 2 подключить онлайн-оплату без миграций.
- **Стартовые курсы:** «Я Целитель» и «Погружение в прошлые жизни» (по методу Долорес Кэннон). В каждом уроке этих курсов — прикреплённый документ .pdf или .docx (см. 3.3).
- **Добавление новых курсов — вручную через админку:** структура (курс → модули → уроки), контент и доступы создаются админом без участия разработчика, чтобы новые курсы можно было заводить в любой момент.

### 3.3 Структура курса и плеер уроков
- Иерархия: **Курс → Модули → Уроки**.
- Страница курса: программа (аккордеон модулей), прогресс-бар, продолжить с последнего урока.
- Страница урока:
  - **собственный видеоплеер платформы** (брендированный, без логотипов сторонних сервисов): HLS-поток из облачного хранилища, рендер через Vidstack / Plyr / hls.js; адаптивное качество (ABR), скорость воспроизведения 0.5–2x, запоминание позиции просмотра;
  - защита контента: подписанные ссылки (token authentication) — поток доступен только авторизованному ученику, прямые ссылки не работают; запрет скачивания на уровне плеера;
  - **текстовое описание урока** под видео: форматированный текст (markdown — заголовки, списки, выделение, ссылки), который админ заполняет при создании урока. Отображается прямо на странице;
  - **прикреплённые файлы (материалы для скачивания):** документы в форматах **.docx (Word)**, .pdf, .pptx, изображения и пр. Ученик видит список файлов с названием, иконкой формата, размером и кнопкой «Скачать». Файлы хранятся в облаке (Bunny Storage), доступ — по подписанной ссылке для авторизованного ученика;
  - *(опционально)* предпросмотр .docx/.pdf прямо в браузере без скачивания (фаза 2; для MVP достаточно скачивания);
  - блок «Конспект» (см. 3.4);
  - кнопки навигации: предыдущий/следующий урок.
- Урок считается **пройденным** только после просмотра видео и **зачтённого ДЗ** (домашнего задания — см. 3.4), если ДЗ обязательно для урока (флаг настраивается).
- **Последовательное открытие уроков:** следующий урок становится доступен только после прохождения предыдущего (просмотр + сдача ДЗ). Это режим по умолчанию для стартовых курсов; флаг строгий/свободный порядок настраивается на уровне курса.

### 3.4 ИИ-наставник: проверка ДЗ ⭐ (ключевая фича)
- После просмотра урока ученик выполняет **домашнее задание (ДЗ)** — пишет краткое содержание усвоенного / отвечает на вопрос урока (текстовое поле, мин. длина настраивается, по умолчанию 200 символов).
- ДЗ отправляется на проверку **ИИ-наставнику (Claude API)**. Агент проверяет усвоение материала **на основе базы знаний курса**, которую заранее загрузил админ (см. ниже), и возвращает JSON:
  - `verdict`: passed / needs_work;
  - `score`: 0–100 (балл за усвоение);
  - `feedback`: краткий вывод (2–3 предложения) по выполненному заданию — что усвоено, что упущено.
- При `needs_work` ученик видит вывод наставника и переписывает ответ (число попыток не ограничено).
- При `passed` урок засчитывается, начисляются баллы (XP + балл за ДЗ идёт в рейтинг).
- Все ответы, баллы и выводы сохраняются; админ видит их в админке.

**Обучение ИИ-агента (вкладка в админке):**
- Отдельная вкладка **«Обучение ИИ-агента»**, где админ загружает материалы по курсу/модулю/уроку (тезисы, эталонные ответы, критерии оценки, при необходимости — файлы). Эта база знаний — основа, по которой агент проверяет ДЗ и выставляет баллы.
- База знаний привязывается к курсу (и опционально уточняется на уровне урока). При проверке ДЗ агент получает релевантный фрагмент базы + ответ ученика.
- Настраиваемые параметры проверки: строгость, проходной балл, формат вывода.

- **Fallback:** если API недоступен — ответ сохраняется со статусом «на проверке», урок временно засчитывается, проверка догоняет фоном.
- Анти-абьюз и безопасность: rate-limit на отправку, защита от prompt injection (см. §6А.8), проверка на копипасту материалов урока.

### 3.5 Геймификация и рейтинг
- **XP (баллы):** начисляются за пройденный урок, за зачтённое ДЗ (бонус по баллу проверки), **за каждый завершённый модуль — дополнительные XP**, за завершение курса. Все значения — в конфиге.
- **Рейтинг (лидерборд):** считается по трём составляющим — **по урокам, по модулям и за выполнение ДЗ**. Таблица учеников по сумме XP — за всё время и за текущий месяц. Видны: место, аватар, имя, XP, стрик. Ученик может скрыть себя из публичного рейтинга.
- **Стрики:** дни подряд с активностью (минимум 1 пройденный урок или сданное ДЗ). Отображаются в профиле и шапке.
- **Достижения — отдельная вкладка:** на странице «Достижения» отображаются **все возможные достижения** (с описанием условий и наградой XP, индикатором прогресса X/Y, категорией и редкостью) и отметка **уже полученных** учеником. Сверху — сводка (получено N из M, XP, прогресс) и фильтры по курсу и категории. Достижения выдаются автоматически по событиям (первый урок, завершение модуля, серия дней, баллы за ДЗ, место в рейтинге и т.п.).
  - **Права:** добавлять/редактировать достижения может **только админ** — кнопка «Добавить достижение» и элементы редактирования видны исключительно в роли админа. Ученик видит только результат (свой прогресс и награды), без каких-либо управляющих элементов.

### 3.6 Профиль ученика
- Аватар, имя, email, дата регистрации.
- Сводка: XP, место в рейтинге, текущий стрик, достижения.
- Список курсов с прогрессом (% и пройденные уроки).
- История ДЗ с баллами и выводами наставника.
- Вкладка «Достижения» (все возможные + полученные, см. 3.5).
- **Подключение Telegram** (привязать/отвязать) и настройки уведомлений (см. 3.9).
- Смена пароля и аватара.

### 3.7 Админ-панель
- **Ручное создание курсов** (курс → модули → уроки) без разработчика: формы с markdown-редактором, загрузка обложек. Заведение новых курсов в любой момент.
- Редактор урока: видео, **текстовое описание (markdown)** и загрузка **файлов-вложений** (.docx, .pdf, .pptx и др.) — несколько файлов на урок, с переименованием и удалением.
- **Вкладка «Обучение ИИ-агента»** (см. 3.4): загрузка базы знаний по курсу/уроку (тезисы, эталоны, критерии, файлы), на основе которой агент проверяет ДЗ и выставляет баллы с кратким выводом. Настройка строгости и проходного балла.
- **Кабинеты учеников:** список учеников, переход в карточку отдельного ученика — его курсы, прогресс, баллы, история ДЗ и выводов наставника, достижения. Поиск, выдача/отзыв доступа к курсам, блокировка.
- **Статистика (дашборд):** активные ученики, завершаемость курсов и модулей, средний балл ДЗ, динамика по рейтингу, расход ИИ-проверок.
- Просмотр ДЗ и выводов агента (фильтр по курсу/ученику/вердикту: все / зачтено / на доработку).
- **Дашборд:** карточки (всего учеников, активные за неделю, средний балл ДЗ, расход ИИ), график активности, завершаемость курсов.
- Все админские экраны собраны в прототипе (раздел «Админ-панель»): Дашборд, Курсы и уроки (дерево курс→модуль→урок + редактор урока с загрузкой видео/описания/файлов), Обучение ИИ-агента (база знаний + параметры), Ученики (таблица + кабинеты + доступы/блокировка), Проверка ДЗ, Достижения. Доступ — только для роли «админ».

### 3.8 Центр знаний (справка)
- Раздел самопомощи: статьи-инструкции «простыми словами», сгруппированные по категориям (Обучение, AI-инструменты, Социальное, Аккаунт).
- Каждая статья: иконка, заголовок, число шагов, короткое описание, переход к соответствующему разделу платформы.
- Поиск по статьям. Внизу — призыв «Не нашёл ответ? Спроси у AI-Наставника».
- Контент статей редактируется админом (как обычные записи в БД).

### 3.9 Интеграция с Telegram ⭐
- Каждый участник может **подключить свой Telegram** к аккаунту (через бота платформы, привязка по одноразовому коду/диплинку). Подключение опционально и управляется в профиле (подключить/отключить в любой момент).
- Через бота **AI-Наставник присылает участнику в Telegram:**
  - **результаты проверки ДЗ** — вердикт (зачтено / на доработку), балл и краткий вывод наставника, сразу после проверки;
  - **оповещения и напоминания:** новый доступный урок/модуль, приближение к новому уровню, поддержка серии (стрик), напоминания вернуться к обучению;
  - **мотивационные сообщения** на основе прогресса (например, раз в 3 дня);
  - уведомления о новых достижениях и изменении места в рейтинге.
- Настройки уведомлений: участник выбирает, какие типы получать; «тихие часы»; полное отключение.
- Технически: Telegram Bot API + вебхук; фоновые задачи (очередь) для рассылки; в сообщениях боту **не передаются секреты**, привязка хранится как `telegram_chat_id` у пользователя. Согласие на этот канал — часть онбординга/настроек.
- Приватность: результаты ДЗ и прогресс отправляются только самому участнику в его привязанный чат; см. §6А (раздел приватности и ИИ).

## 5. Технологический стек

### 5.1 Основной
- **Next.js 15 (App Router) + TypeScript** — фронт и бэк (API routes/server actions) в одном проекте: меньше инфраструктуры, идеально под соло-разработку с Claude Code.
- **Tailwind CSS + shadcn/ui** — быстрая сборка UI (стиль определим отдельно, см. §8).
- **PostgreSQL + Prisma ORM** — типобезопасная работа с БД, простые миграции.
- **Auth.js (NextAuth v5)** — email+пароль (credentials) и задел под Google OAuth.
- **Anthropic SDK (`@anthropic-ai/sdk`)** — проверка конспектов через Claude API (вызовы только с сервера, ключ в env). Документация: https://docs.claude.com/en/api/overview
- **Resend** (или SMTP) — письма: подтверждение email, сброс пароля.
- **Telegram Bot API** (`grammY` или `node-telegram-bot-api`) — бот-наставник: привязка участников, отправка результатов проверки ДЗ и оповещений. Рассылка через очередь/фоновые задачи; токен бота в env.
- **Zod** — валидация входных данных.

### 5.2 Хостинг приложения и сервер (бюджетный, мировая аудитория)

Ключевая идея: **тяжёлый трафик (видео) идёт через CDN видеохостинга, а не через сервер приложения**. Поэтому для 100–150 учеников мощный сервер не нужен — приложение обслуживает только страницы и API.

**Вариант A (рекомендуется для старта): serverless, $0/мес**
- **Vercel (Hobby)** — деплой Next.js, глобальная edge-сеть (быстро из любой точки мира), автомасштабирование, HTTPS из коробки.
- **Neon (free tier)** — managed PostgreSQL, автобэкапы.
- Плюсы: ноль администрирования, нулевая цена на этом масштабе. Минусы: лимиты free-tier (при росте — Vercel Pro $20/мес).

**Вариант B (свой VPS, полный контроль): ~€4–6/мес**
- **Hetzner Cloud CX23** (2 vCPU, 4 GB RAM, NVMe, 20 TB трафика) — с запасом хватает на Next.js + PostgreSQL в Docker для 150 пользователей.
- Деплой через **Coolify/Dokploy** (self-hosted PaaS) + Cloudflare (бесплатно) перед сервером для CDN статики и защиты.
- Плюсы: всё своё, предсказуемая цена. Минусы: администрирование (обновления, бэкапы БД) на тебе.

Решение: начинаем с варианта A; переезд на VPS возможен в любой момент (Next.js + Postgres переносимы).

### 5.3 Видео: облачное хранилище + свой плеер

Требование: видео хранятся в облаке, проигрываются через собственный плеер платформы, скачивание и шаринг ссылок исключены, аудитория по всему миру.

**Рекомендация: Bunny Stream** — закрывает всё одним сервисом:
- хранилище + автоматическая транскодация в HLS (несколько качеств, ABR) включена в цену;
- глобальный CDN (119+ точек присутствия) — быстрый старт видео в любой стране;
- **Token Authentication / signed URLs** — поток отдаётся только по подписанной ссылке, которую генерирует наш бэкенд для авторизованного ученика; срок жизни ссылки — минуты;
- плеер: используем свой (Vidstack/hls.js), Bunny отдаёт чистый HLS-манифест.

Ориентировочная цена: хранение ~$0.01/GB/мес, доставка от ~$0.005–0.01/GB. Прогноз для нашего масштаба (≈100 GB видео, 150 учеников, активный просмотр): **$10–30/мес**.

Альтернативы (отклонены): Cloudflare Stream — $5/1000 мин хранения + $1/1000 мин доставки; на нашем объёме просмотров выходит в разы дороже Bunny. AWS S3 + CloudFront — дороже по egress и требует самостоятельной настройки транскодации.

Реализация в коде: модуль `video-provider` с интерфейсом `getSignedStreamUrl(lessonId, userId)` — провайдер абстрагирован, замена сервиса не трогает остальной код.

**Плеер vs хранилище (чтобы не путать):** *хранилище* — это Bunny (сторонний сервис, где лежит видео). *Плеер* — наш собственный React-компонент в фирменном стиле на базе открытой библиотеки **hls.js / Vidstack** (это код внутри проекта, не сторонний сервис и без чужого брендинга). Связка: видео в Bunny → подписанная HLS-ссылка → наш плеер проигрывает поток на странице урока (запоминание позиции «продолжить с…», скорость 0.5–2×, отметка «просмотрено»).

### 5.5 Где что хранится (сводно)

Принцип: **тяжёлое и бинарное — в облачное хранилище с CDN; лёгкое и структурированное — в базу данных.**

| Данные | Где хранятся | Как ученик получает |
|---|---|---|
| Видео уроков | **Bunny Stream** (облако + CDN) | подписанная HLS-ссылка с коротким TTL → собственный плеер |
| Файлы-вложения (.pdf/.docx/.pptx, обложки, аватары) | **Bunny Storage** (облако + CDN) | подписанная ссылка на скачивание после проверки доступа |
| Тексты ДЗ + вердикты, баллы, выводы ИИ | **PostgreSQL** (таблица `homework`) | отображаются в уроке/профиле/админке |
| Прогресс, XP, достижения, рейтинг, enrollment | **PostgreSQL** | через приложение |
| Метаданные файлов и видео (ссылка, тип, размер) | **PostgreSQL** (`lesson_files`, `lessons`) | — |

**Важно (независимость от сервиса):** в коде доступ к видео и файлам идёт через абстракции `VideoProvider` / файловый провайдер (`lib/providers/`). Имя «Bunny» живёт только внутри одного файла-адаптера. Поэтому смена хостинга (например, на S3 или Supabase Storage) — это правка одного адаптера; страницы уроков, плеер и логика скачивания остаются нетронутыми. Bunny здесь — рекомендованная реализация по умолчанию, а не жёсткая зависимость.

Env-переменные: `DATABASE_URL`, `AUTH_SECRET`, `ANTHROPIC_API_KEY`, `RESEND_API_KEY`, `BUNNY_API_KEY`, `BUNNY_LIBRARY_ID`, `BUNNY_TOKEN_AUTH_KEY`, `TELEGRAM_BOT_TOKEN`.

## 6А. Безопасность и защита персональных данных

Платформа хранит персональные данные учеников (email, имя, прогресс обучения, тексты конспектов — могут содержать личные размышления). Все пункты ниже — обязательная часть MVP, а не «фаза 2».

### 6А.1 Классификация данных
- **Идентифицирующие (PII):** email, имя, аватар, IP/устройство в логах.
- **Чувствительные по смыслу:** тексты конспектов и рефлексии (личные мысли ученика), история активности.
- **Секреты:** хэши паролей, токены сессий, API-ключи сервисов.
- Принцип минимизации: собираем только то, что нужно для обучения; платёжные данные не храним (оплаты на платформе нет).

### 6А.2 Аутентификация и пароли
- Пароли хранятся только в виде хэша **argon2id** (или bcrypt, cost ≥ 12). Открытые пароли нигде не логируются и не сохраняются.
- Требования к паролю: минимум 8–10 символов, проверка по списку утёкших (haveibeenpwned k-anonymity API) — опционально.
- Защита от перебора: rate-limit и экспоненциальная задержка на попытки входа, временная блокировка после N неудач, CAPTCHA при подозрении.
- Сессии: HttpOnly + Secure + SameSite=Lax cookie, короткий TTL access-токена + ротация refresh-токена, инвалидизация при смене пароля.
- Сброс пароля и подтверждение email — одноразовые токены с коротким сроком жизни (15–30 мин), привязанные к пользователю.
- **2FA (TOTP) обязательна для роли «Админ»**; для учеников — опционально (фаза 2).
- **Защита от перечисления пользователей (enumeration):** формы входа, регистрации и сброса пароля дают одинаковый ответ независимо от того, существует email или нет («если аккаунт есть — письмо отправлено»). Сообщения об ошибках не раскрывают, что именно неверно.

### 6А.3 Авторизация и контроль доступа
- Проверка прав на **каждом** защищённом запросе (middleware), а не только в UI.
- Проверка владения ресурсом (ownership): ученик видит/меняет только свои данные; нельзя получить чужой конспект или урок закрытого курса подменой ID.
- Доступ к уроку выдаётся только при наличии активного `enrollment`.
- Разделение ролей (ученик / куратор / админ) с принципом наименьших привилегий. Админ-маршруты — отдельная зона с обязательной 2FA.
- **Назначение ролей только сервером:** роль «админ»/«куратор» нельзя получить через публичную регистрацию или подменой запроса — выдаётся вручную в БД/существующим админом. Новые регистрации всегда создаются с ролью «ученик».

### 6А.4 Передача и хранение данных
- **Только HTTPS/TLS** (HSTS включён); HTTP редиректит на HTTPS.
- Шифрование «в покое»: БД у managed-провайдера (Neon/Supabase) шифруется на стороне хранилища; бэкапы тоже зашифрованы.
- Доступ к БД — только из приложения по защищённому соединению, БД не торчит в открытый интернет; доступ по allowlist/частной сети.
- Тексты ДЗ в БД (PostgreSQL, таблица `homework`) хранятся как обычный текст (нужны для отображения и проверки); защита — на уровне доступа и шифрования хранилища.

### 6А.5 Управление API-ключами и секретами

**Главный принцип:** ключи (Anthropic, Bunny, Telegram, БД, почта) — это серверные секреты. Они **никогда не покидают сервер** и недоступны из браузера. «Зашифровать ключ в коде» нельзя (тогда расшифровать сможет и злоумышленник) — защита строится на изоляции и хранении в защищённом хранилище.

**Где и как хранятся:**
- Все ключи — только в **переменных окружения хостинга / менеджере секретов** (Vercel Environment Variables, либо Doppler/Infisical/HashiCorp Vault). Провайдер хранит их **зашифрованными at rest**; в приложение они попадают только в рантайме.
- В репозитории секретов **нет никогда** — в git только `.env.example` без значений. `.env` в `.gitignore`. Сканер секретов (gitleaks / GitHub secret scanning) в CI блокирует случайный коммит ключа.
- Разные наборы ключей для dev / staging / prod.

**Изоляция от клиента (ключевое против кражи):**
- Все вызовы внешних API (Claude, Bunny, Telegram, почта) идут **только из серверного кода** (server actions / API routes). Ключ физически не входит в JS-бандл, который отдаётся браузеру.
- В Next.js это гарантируется правилом: секреты — без префикса `NEXT_PUBLIC_` (такие переменные попадают в клиент); серверные модули с ключами не импортируются в клиентские компоненты.
- Браузер обращается к нашему серверу, сервер — к Claude/Bunny. Пользователь и его devtools ключ не видят. Видео и файлы отдаются через **подписанные ссылки с коротким TTL** (см. 6А.7), а не через раздачу ключа.

**Если секрет всё же нужно хранить в нашей БД** (например, токен бота на пользователя — в MVP не требуется): хранить в **зашифрованном виде** (AES-256-GCM), ключ шифрования — в менеджере секретов, а не в БД. `telegram_chat_id` секретом не является.

**Ротация и реакция на утечку:**
- Возможность **ротации ключей без простоя** (обновил в менеджере секретов → передеплой). Плановая ротация раз в N месяцев.
- При подозрении на утечку: немедленно **отозвать/перевыпустить** ключ в консоли провайдера (Anthropic/Bunny/Telegram), инвалидировать сессии, проверить логи доступа.
- **Ограничение ущерба:** ключи с минимальными правами (least privilege); где возможно — лимиты расхода и алерты на аномальный объём (защищает от «слива» через скомпрометированный ключ и от неожиданного счёта).

**Защита от косвенных утечек:**
- Ключи и токены **не логируются** и не попадают в тексты ошибок, отдаваемых пользователю (см. 6А.9).
- Защита от SSRF: серверные функции не дёргают произвольные URL по пользовательскому вводу (чтобы нельзя было заставить сервер «принести» свои env/секреты).
- Доступ к консолям Vercel/Neon/Bunny/Anthropic — с обязательной 2FA и минимумом людей (см. 6А.12).

### 6А.6 Защита от типовых атак (OWASP)
- **SQL-инъекции:** только параметризованные запросы через Prisma ORM, без сырых строк с конкатенацией.
- **XSS:** экранирование пользовательского ввода при выводе; конспекты рендерятся как текст (или санитизированный markdown через DOMPurify), без вставки сырого HTML.
- **CSRF:** SameSite-cookie + CSRF-токены на мутациях.
- **Валидация ввода:** все входные данные через Zod-схемы на сервере (длина, тип, формат email).
- **Rate limiting:** на вход, регистрацию, сброс пароля, отправку конспектов и вызовы ИИ — защита от абьюза и от роста счёта за Claude API.
- **Защита заголовками:** CSP, X-Frame-Options (анти-кликджекинг), X-Content-Type-Options, Referrer-Policy.
- **Загрузка файлов** (аватары, обложки, **вложения уроков .docx/.pdf/.pptx**): белый список разрешённых типов, проверка **реального MIME** (по содержимому, не по расширению), лимит размера, хранение в облаке (Bunny Storage) вне корня приложения, без исполняемых файлов. Картинки переэнкодируются с удалением EXIF-метаданных. При отдаче файлов — заголовки `Content-Disposition: attachment` и `X-Content-Type-Options: nosniff` (файл скачивается, а не исполняется в браузере). Желательно антивирусная проверка загрузок (напр., ClamAS/внешний скан) — минимум для админских загрузок.

### 6А.7 Защищённый контент (видео и файлы-вложения)
- **Видео:** доступ к потоку — только по подписанной ссылке (token authentication Bunny) с коротким TTL, генерируемой бэкендом для авторизованного ученика с активным доступом. Защита от хотлинкинга и прямого скачивания; ссылки не кэшируются на клиенте надолго.
- **Файлы-вложения (.docx/.pdf и пр.):** скачивание тоже **только по подписанной ссылке с коротким TTL** и проверкой, что у ученика есть активный `enrollment` на курс этого урока. Прямой публичный URL файла недоступен. Это закрывает утечку платных материалов через переслать-ссылку.

### 6А.8 ИИ-проверка конспектов (приватность и безопасность)
- В Claude API отправляется минимум: текст конспекта + тезисы урока. **Без** email и имени ученика (только обезличенный ID при необходимости).
- Защита от **prompt injection**: текст ученика передаётся как данные в отдельном поле, системный промпт чётко отделяет инструкции от пользовательского контента; результат парсится строго как ожидаемый JSON, любые «команды» внутри конспекта игнорируются.
- Лимит длины и частоты запросов; обработка ошибок API без раскрытия внутренних деталей пользователю.

### 6А.9 Логирование, мониторинг, аудит
- В логи **не попадают** пароли, токены, полные email (маскирование), тексты конспектов.
- Аудит-журнал значимых действий: вход, смена пароля, выдача/отзыв доступа админом, удаление аккаунта.
- Мониторинг ошибок (Sentry free) и аптайма (UptimeRobot); алерты на всплески 4xx/5xx и на аномальный расход Claude API.

### 6А.10 Резервное копирование и восстановление
- Автоматические ежедневные бэкапы БД (входят в Neon/Supabase), хранение ≥ 7 дней, шифрование.
- Периодическая проверка восстановления из бэкапа (хотя бы вручную перед крупными релизами).

### 6А.11 Правовое и приватность (важно для мировой аудитории)
- **Политика конфиденциальности** и **Пользовательское соглашение** — обязательны при регистрации (чекбокс согласия на обработку ПД).
- Право на доступ и удаление данных: ученик может запросить экспорт и удаление аккаунта (право «быть забытым» — учитывая GDPR, раз аудитория глобальная).
- **Механика удаления:** при удалении аккаунта каскадно удаляются/обезличиваются связанные данные (конспекты, прогресс, достижения). Оговорка: данные могут оставаться в зашифрованных бэкапах до истечения срока их хранения (≤ N дней) — это указывается в политике.
- **Доступ персонала к конспектам:** админ/куратор может читать тексты конспектов учеников (для проверки обучения). Это раскрывается в политике конфиденциальности и фиксируется в аудит-журнале (кто и когда смотрел).
- Прозрачность: указать, что конспекты обрабатываются ИИ (Anthropic) для проверки; ссылка на политику.
- Срок хранения данных и порядок удаления неактивных аккаунтов — задокументировать.
- Cookie-уведомление (используются только функциональные cookie сессии — это упрощает согласие).

### 6А.12 Безопасность разработки и зависимостей
- `npm audit` / Dependabot — отслеживание уязвимостей в пакетах, регулярные обновления.
- Защита ветки `main`, ревью изменений, отсутствие секретов в коммитах (gitleaks/secret-scanning).
- Принцип least privilege для доступов к Vercel/Neon/Bunny-аккаунтам, 2FA на этих сервисах.

### 6А.13 Реагирование на инциденты
- План на случай утечки: отозвать/ротировать ключи и сессии, оценить объём, уведомить затронутых пользователей.
### 6А.14 Проверка перед запуском (security gate)
- Чек-лист перед продакшеном: все секреты в env и ротированы, 2FA у админов включена, HTTPS/HSTS работает, заголовки безопасности отдаются, rate-limit активен на чувствительных эндпоинтах, тестовые/seed-аккаунты удалены.
- Базовая проверка: прогон `npm audit`, ручной тест ownership (попытка открыть чужой ресурс по ID), проверка, что приватные видео/файлы недоступны без авторизации.
- Желательно: внешнее сканирование (OWASP ZAP / простой pentest) до открытия доступа реальным ученикам.
